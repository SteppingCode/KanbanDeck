# KanbanDeck

1. Overview
    1) Веб-приложение для визуального управления задачами на канбан-доске.
    2) Реализовано на Django с интерфейсом drag-and-drop для перемещения задач между колонками.
    3) Целевая аудитория: индивидуальные пользователи, студенты и команды, которым нужна простая доска задач.

2. Problem/Context
    1) Перемещение задач между статусами часто выполняется вручную и неудобно.
    2) Необходим простой интерфейс для создания и перемещения карточек по колонкам.
    3) Требуется хранение данных в единой базе с быстрым доступом из браузера.

3. Goals
    1) Сделать базовую канбан-доску с досками, колонками и задачами.
    2) Обеспечить добавление новых задач в колонку.
    3) Обеспечить перемещение задач между колонками с сохранением в базе данных.

4. Non-Goals
    1) На текущем этапе не реализовано создание досок через UI.
    2) Нет редактирования или удаления задач из интерфейса.
    3) Не предусмотрено аутентификация, права доступа и мобильная версия.

5. Solution (High-Level Design)
    Пользователь → Веб-браузер → Django-приложение → SQLite.

6. Detailed Design
    1. Backend:
        - Django приложение `deck`.
        - Модели: `Board`, `Column`, `Task`.
        - URL-маршруты:
            - `/board/<board_id>/` — отображение доски.
            - `/column/<column_id>/add_task/` — добавление задачи в колонку.
            - `/move_task/` — перемещение задачи между колонками.
    2. Frontend:
        - Шаблон `templates/board.html`.
        - Задачи выводятся в колонках.
        - Для drag-and-drop используется библиотека SortableJS.
    3. Data Flow:
        - Пользователь перетаскивает карточку.
        - JavaScript отправляет POST-запрос на `/move_task/`.
        - Сервер обновляет `column_id` задачи и сохраняет изменения в базе.

7. Data Model
    1) `Board`: имя доски.
    2) `Column`: колонка принадлежит доске, имеет имя и порядок.
    3) `Task`: задача принадлежит колонке, имеет заголовок и порядок.

8. Alternatives
    1) SQLite используется для простоты разработки и локального прототипа.
    2) В дальнейшем можно перейти на PostgreSQL для продакшн-развертывания.

9. Trade-offs
    1) Простота текущей реализации на Django позволяет быстро прототипировать, но ограничивает функционал.
    2) Нет фронтенд-редактирования задач, что снижает гибкость, но упрощает MVP.

10. Scaling
    1) Переход на внешнюю базу данных (PostgreSQL).
    2) Добавление API для создания досок и управления колонками.
    3) Вынесение статики и шаблонов на CDN / отдельный фронтенд.

11. Reliability
    1) Сервер хранит состояние в SQLite-файле `db.sqlite3`.
    2) При интенсивной нагрузке база SQLite становится узким местом.
    3) Для стабильности требуется отдельная СУБД и резервное копирование.

12. Metrics
    1) Latency: время ответа сервера на простые POST/GET-запросы.
    2) Error rate: корректная обработка невалидных запросов (например, пустых task_id).
    3) Uptime: доступность проекта при локальном запуске.
    4) Load time: время загрузки страницы доски и инициализации SortableJS.

13. Open questions
    1) Добавить создание досок и колонок в UI?
    2) Стоит ли реализовать редактирование и удаление задач?
    3) Нужны ли теги, сроки и назначенные исполнители для карточек?
